OpenAI Responses API Now Supports WebSockets for Faster Agentic Workflows

OpenAI has added WebSocket support to the Responses API, cutting end-to-end latency for agentic workflows by up to 40% by eliminating the overhead of repeated HTTP connections.

OpenAI Newsroom 2026-04-22 21:57 UTC Key: openai-newsroom::identity::speeding-up-agentic-workflows-with-websockets Confidence: strong Mode: claude

Article body

Under the hood, tools like Codex make dozens of back-and-forth requests per task—determining the model's next action, running a tool, sending results back, and repeating. As inference speed has climbed, that cumulative API overhead became the new bottleneck, slowing down even the fastest models.

To solve this, OpenAI switched from sequential HTTP calls to a persistent WebSocket connection. The server caches conversation state—including prior input/output items, tool definitions, and pre-rendered tokens—so follow-up requests reuse that context instead of reprocessing the full history each time. Safety classifiers and validators now operate only on new input, and post-inference tasks like billing run concurrently with subsequent requests.

The results were immediate. GPT-5.3-Codex-Spark hit its 1,000 tokens-per-second target with bursts up to 4,000 TPS. Codex ramped the majority of its Responses API traffic onto WebSocket mode within weeks of launch. External integrators felt the gains too: Vercel's AI SDK saw up to 40% latency reduction, Cline's multi-file workflows ran 39% faster, and Cursor users saw up to 30% improvement.

Why this matters

  • Agentic AI tools like Codex, Cursor, and Cline are now meaningfully faster without changing their underlying model calls.
  • The persistent connection design avoids rebuilding conversation state from scratch on every request, directly addressing the growing gap between inference speed and API overhead.
  • The two-month sprint from concept to production highlights how transport-layer changes can unlock gains that pure model improvements cannot alone.
