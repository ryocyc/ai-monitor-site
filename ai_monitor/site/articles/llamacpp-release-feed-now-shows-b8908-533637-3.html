llama.cpp release b8908 patches heap-buffer-overflow in server

The llama.cpp project has shipped a security fix for a heap-buffer-overflow triggered by a negative n_discard value in the server component, closing a high-severity vulnerability with CVSS 8.8.

GitHub llama.cpp Releases 2026-04-23 23:18 UTC Key: github-llamacpp-releases::identity::b8908 Confidence: strong Mode: claude

Article body

Release b8908, published on April 23, addresses a heap-buffer-overflow vulnerability in llama.cpp's server component. The flaw lives in the update_slots() context-shift loop, where a negative n_discard value passed in client JSON can cause out-of-bounds memory access. The issue has been assigned CVE-2026-21869 and classified under CWE-787, with a CVSS score of 8.8 — placing it in the High severity range.

The fix applies a two-part mitigation. First, the root cause is patched directly in the update_slots() loop. Second, as a hardening measure at the JSON parse boundary, n_discard is clamped to non-negative values at ingress. The advisory notes that n_discard=0 already triggers an auto-discard fallback (n_left/2), so clamping to 0 is safe and does not alter expected behavior. The fix references GitHub Security Advisory GHSA-8947-pfff-2f3c and is co-authored by Georgi Gerganov.

The release ships pre-built binaries across a broad range of platforms: macOS (Apple Silicon, KleidiAI-enabled Apple Silicon, Intel), iOS, multiple Linux distributions and CPU architectures (x64, arm64, s390x) with support for Vulkan, ROCm 7.2, OpenVINO, and SYCL, plus Android arm64, Windows (CPU, CUDA 12/13, Vulkan, SYCL, HIP), and openEuler builds for both x86 and aarch64. Note that some UI elements on the release page failed to load fully, so asset links were not accessible at capture time.

Why this matters

  • A heap-buffer-overflow (CWE-787) can be exploited to read or write adjacent memory, potentially allowing an attacker to crash the server or achieve code execution.
  • The CVSS 8.8 score means the vulnerability is reachable over the network without special privileges, making it a realistic risk for any llama.cpp server exposed to untrusted input.
  • The fix is delivered as a single tagged release with platform-native binaries for every major OS and accelerator backend, lowering the barrier to patching.
