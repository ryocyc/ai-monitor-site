Speeding Up Agentic Workflows with WebSockets in the Responses API

OpenAI has added WebSocket support to the Responses API, cutting agentic workflow latency by 40% by replacing a series of synchronous HTTP calls with a single persistent connection.

OpenAI Newsroom 2026-04-22 21:57 UTC Key: openai-newsroom::identity::speeding-up-agentic-workflows-with-websockets Confidence: moderate Mode: claude

Article body

OpenAI's engineering team has shipped WebSocket support in the Responses API, a change designed to eliminate the accumulated overhead that builds up when an AI agent makes dozens of back-and-forth requests during a single task. In a post published April 22, engineers Brian Yu and Ashwin Nathan explain that as inference speeds have improved, the cost of setting up and tearing down a new HTTP connection for each request has become the main bottleneck for latency-sensitive workloads like Codex's coding agent loop.

The core shift is architectural: instead of sending the full conversation history with every follow-up request, the server now keeps a connection-scoped, in-memory cache of prior response state—including input/output items, tool definitions, and rendered tokens—across the lifetime of the WebSocket connection. Clients continue using the same `response.create` call and pass a `previous_response_id` to pull cached state, meaning developers don't need to rewrite their existing API integration. Safety classifiers and request validators were also updated to process only new input rather than re-evaluating the full history on each turn.

Early results from an alpha with coding-agent partners were positive enough to push to general availability. Codex quickly moved the majority of its Responses API traffic onto WebSocket mode. External integrations confirmed the gains: Vercel's AI SDK and Cline's multi-file workflows both saw roughly 40% latency reductions, while OpenAI models running in Cursor improved by up to 30%. For the fast coding model GPT-5.3-Codex-Spark, the team hit its 1,000 tokens-per-second target and observed bursts reaching 4,000 TPS.

Why this matters

  • Agentic workflows—where an AI makes repeated tool calls to complete a task—generate many sequential API requests, and that overhead had become the limiting factor as inference speeds increased.
  • Developers can adopt WebSocket mode without changing their API input/output shapes, since the interface remains based on `response.create` and `previous_response_id`.
  • Third-party integrations including Vercel AI SDK, Cline, and Cursor are already reporting 30–40% latency improvements in production workloads, suggesting broad applicability beyond Codex.
