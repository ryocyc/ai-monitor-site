Speeding up Agentic Workflows with WebSockets in the Responses API

OpenAI has added WebSocket support to its Responses API, cutting end-to-end latency for agentic AI workflows by up to 40 percent.

OpenAI Newsroom 2026-04-22 21:57 UTC Key: openai-newsroom::identity::speeding-up-agentic-workflows-with-websockets Confidence: strong Mode: claude

Article body

When you ask Codex to fix a bug, it scans your codebase, reads relevant files, makes edits, and runs tests—meaning dozens of back-and-forth API requests per task. As inference speeds have climbed, that cumulative API overhead became the new bottleneck. OpenAI's solution: a persistent WebSocket connection that caches conversation state in memory, so only new information needs to be validated on each step instead of re-processing the entire history every time. The result is a roughly 40 percent improvement in end-to-end agentic rollout speed.

The key technical shift is that the server now keeps a connection-scoped, in-memory cache of prior response state—including the previous response object, input/output items, tool definitions, and rendered tokens. When a follow-up request carries a `previous_response_id`, the API fetches that cached state rather than rebuilding from scratch. Developers keep the familiar `response.create` shape and simply add `previous_response_id`, so no API contract rewrite is required. The WebSocket transport lets the server send a `response.done` event after a tool call is sampled, the client executes the tool and sends back a `response.append` event, and the model continues—treating the local tool call much like a hosted remote tool call.

In production, Codex quickly moved the majority of its Responses API traffic onto WebSocket mode, and GPT-5.3-Codex-Spark hit the 1,000 tokens-per-second target with bursts reaching 4,000 TPS. Third-party integrations showed the gains translating downstream: Vercel's AI SDK latency dropped by up to 40 percent, Cline's multi-file workflows ran 39 percent faster, and Cursor saw OpenAI models become up to 30 percent faster. OpenAI calls WebSocket mode one of the most significant new capabilities in the Responses API since its launch in March 2025.

Why this matters

  • Agentic loops like Codex were bottlenecked by cumulative API overhead as inference speed improved, making multi-step tasks feel sluggish even when the model itself was fast.
  • The persistent connection approach eliminates redundant work across an entire agent rollout—pre-inference, validation, and token rendering are done once, not per step.
  • Because the API surface stays unchanged for developers, adoption is low-friction: existing `response.create` code just adds a `previous_response_id` parameter over WebSocket.
