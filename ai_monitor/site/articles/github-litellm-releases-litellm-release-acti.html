LiteLLM Adds Cosign Docker Image Signing for All Releases

The BerriAI team has enabled cryptographic signing of all LiteLLM Docker images using cosign, giving users a verifiable chain from image to source.

GitHub LiteLLM Releases 2026-04-22 15:01 UTC Key: github-litellm-releases::identity::github-litellm-releases-litellm-release-acti Confidence: strong Mode: claude

Article body

All LiteLLM Docker images published to ghcr.io/berriai/litellm are now signed with cosign, the CNCF-born tool for container image signing and verification. The same signing key introduced at commit 0112e53 covers every release, including nightly and stable tags. Users can verify an image by pinning the commit hash rather than relying on a mutable tag, which the project calls the strongest approach since commit hashes are cryptographically immutable.

Two verification paths are documented. The recommended method pulls the public key directly from the pinned commit hash URL and runs `cosign verify`. The convenience method instead fetches the key from a release tag URL; the project notes this is easier to read but depends on tag protection rules. Both methods confirm that cosign claims were validated and signatures matched the specified public key.

Version v1.83.11-nightly (released 2026-04-22) is the latest nightly build, while v1.83.10-nightly notably dropped support for Python 3.9, raising the minimum to Python 3.10 in line with Python's end-of-life schedule for the 3.9 branch. The changelog also includes a range of proxy improvements, Bedrock Mantle endpoint support, new UI features, and tooling updates.

Why this matters

  • Container signing closes a supply-chain gap: users can now cryptographically confirm the image they pull matches what the LiteLLM team actually published, not a tampered substitute.
  • Pin-to-commit verification removes any reliance on mutable tag state, offering a reproducible trust anchor independent of GitHub's tag protection mechanisms.
  • The v1.83.10-nightly minimum Python upgrade to 3.10 means any existing Python 3.9 deployments will need updating before upgrading.
