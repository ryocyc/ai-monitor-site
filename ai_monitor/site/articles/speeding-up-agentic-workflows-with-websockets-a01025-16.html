OpenAI Responses API Now Supports WebSockets for Faster Agentic Workflows

OpenAI has added WebSocket support to its Responses API, cutting end-to-end agentic rollout latency by 40% by replacing a series of synchronous HTTP calls with a single persistent connection.

OpenAI Newsroom 2026-04-22 21:57 UTC Key: openai-newsroom::identity::speeding-up-agentic-workflows-with-websockets Confidence: strong Mode: claude

Article body

As OpenAI's inference speed climbed toward 1,000 tokens per second with models like GPT‑5.3‑Codex‑Spark, the cumulative overhead of repeated HTTP handshakes became the new bottleneck. Each tool call in a Codex agent loop — read a file, apply a patch, run a test — had previously required a fresh connection, re-validation of the full conversation history, and re-tokenization of context already seen. OpenAI addressed this by redesigning the transport layer: instead of closing the connection after every response, a WebSocket keeps state cached in memory for the lifetime of the session. When a follow‑up request carries a `previous_response_id`, the server pulls prior response objects, tool definitions, and rendered tokens from that connection-scoped cache rather than rebuilding them from scratch. Safety classifiers and validators also process only new input, skipping redundant checks against unchanged history.

The first prototype, built by a single engineer overnight, modeled an entire agentic rollout as a single long-running Response, with the inference loop pausing after each tool call and resuming when the client sent back results via WebSocket. While effective, that prototype introduced a novel and more complex API shape. For the shipped version, OpenAI reverted to a familiar interface — developers continue using `response.create` with the same request body, adding only a `previous_response_id` to carry context forward. The underlying persistent connection handles the rest invisibly, allowing teams to adopt WebSocket mode without rewriting their existing API integration.

After a two-month sprint, OpenAI ran an alpha with key coding agent partners before opening WebSocket mode more broadly. The results landed quickly: Codex moved the majority of its Responses API traffic onto WebSockets, Vercel's AI SDK saw up to 40% latency reductions, Cline's multi-file workflows ran 39% faster, and Cursor reported up to 30% speedups with the latest models. OpenAI describes WebSocket mode as one of the most significant new capabilities in the Responses API since its March 2025 launch, noting that as inference hardware accelerates, the surrounding service layer must speed up in parallel to deliver those gains to users.

Why this matters

  • Agents that make dozens of tool calls per task previously paid a latency tax on every round-trip; persistent WebSocket connections eliminate redundant validation, tokenization, and routing work across an entire session.
  • Cerebras-powered models like GPT‑5.3‑Codex‑Spark now sustain over 1,000 tokens per second in production, and WebSocket mode ensures API overhead no longer negates those inference speed gains.
  • Developers can adopt WebSocket mode without changing their `response.create` call shape, making the upgrade incremental rather than a full rewrite.
