LiteLLM v1.83.12-Nightly Adds Cosign Image Signing and Moonshot/kimi-k2.6 Model

LiteLLM's latest nightly release introduces cryptographic image signing for all Docker images via cosign, alongside new model and routing additions.

GitHub LiteLLM Releases 2026-04-24 01:24 UTC Key: github-litellm-releases::identity::v18312-nightly Confidence: strong Mode: claude

Article body

The BerriAI team published LiteLLM v1.83.12-nightly on April 23rd, 2026, bringing cosign-based Docker image signature verification to all LiteLLM images published to ghcr.io. Each release is signed with a single key rooted in commit 0112e53. Operators can verify images using either the pinned commit hash — described in the release notes as the strongest approach since a hash is cryptographically immutable — or the release tag, which relies on repository tag protection rules.

The release also adds the Moonshot/kimi-k2.6 model to the LiteLLM model registry and ships a new adaptive routing feature. Several infrastructure and stability improvements are included: restoration of a pre-uv Prisma cache path in Docker, migration of more CI jobs from CircleCI to GitHub Actions, a fix for Bedrock guardrail spend logs in proxy hook mode, wildcard order fallback logic for the router, and stabilization work targeting spend accuracy tests and Redis buffer data-loss paths.

The changelog notes six contributors to this release, with the full asset list accessible via the GitHub release page. Users pulling LiteLLM via Docker are encouraged to verify the signature before running the image, using either the recommended commit-hash method or the tag-based convenience method documented in the release.

Why this matters

  • Cosign signing gives operators a cryptographic chain of trust for LiteLLM Docker images, reducing the risk of running tampered or unofficial containers.
  • Adding Moonshot/kimi-k2.6 to the model registry expands LiteLLM's coverage of Chinese AI providers, enabling consistent API access through LiteLLM's unified interface.
  • The adaptive routing feature suggests LiteLLM is moving toward intelligent request distribution, which could improve reliability and cost efficiency in multi-model deployments.
