OpenAI Responses API Now Supports WebSockets for Faster Agentic Workflows

OpenAI has added WebSocket support to its Responses API, cutting end-to-end latency for agentic AI workflows by up to 40% and enabling inference speeds that were previously obscured by API overhead.

OpenAI Newsroom 2026-04-22 21:57 UTC Key: openai-newsroom::identity::speeding-up-agentic-workflows-with-websockets Confidence: strong Mode: claude

Article body

When an AI coding agent like Codex fixes a bug, it fires dozens of back-and-forth requests to the Responses API — each one carrying the full conversation history, re-validating context, and re-rendering tokens that changed little or not at all. As model inference has grown faster, that accumulated API overhead became the new bottleneck. OpenAI's engineering team set out to close that gap, and their solution is WebSocket mode: a persistent connection to the Responses API that caches reusable state — prior input and output items, tool definitions, and previously rendered tokens — for the lifetime of the connection, so follow-up requests only send what's new.

The design deliberately keeps the familiar API shape developers already know. Rather than adopting a new interaction mode, callers continue using `response.create` with the same request body; the only addition is a `previous_response_id` that lets the server pull cached state from the in-memory connection-scoped cache. This enables targeted optimizations: safety classifiers and request validators now process only new input rather than replaying the full history, rendered tokens are cached and appended to rather than recomputed, and non-blocking postinference work like billing can overlap with subsequent requests.

After a two-month alpha with key coding agent startups, WebSocket mode shipped broadly. Codex moved the majority of its Responses API traffic onto it almost immediately, and GPT-5.3-Codex-Spark hit the 1,000 tokens-per-second target — with bursts up to 4,000 TPS in production traffic. Third-party integrators felt the shift too: Vercel's AI SDK saw latency drop by up to 40%, Cline's multi-file workflows ran 39% faster, and Cursor reported up to 30% speedups on OpenAI models.

Why this matters

  • As model inference speeds approach thousands of tokens per second, API overhead becomes the dominant latency source — WebSockets address this at the transport layer without requiring developers to rewrite their API integration.
  • The 40% end-to-end latency reduction for agentic workflows translates directly to faster, more responsive AI coding assistants and agentic tools used by developers daily.
  • The persistent connection and in-memory state caching approach is portable in concept, suggesting similar transport-level optimizations could arrive in other LLM API platforms.
