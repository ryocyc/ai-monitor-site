OpenAI Responses API Now Supports WebSockets for Faster Agentic Workflows

OpenAI has added WebSocket support to its Responses API, reducing end-to-end latency in agentic workflows by up to 40 percent.

OpenAI Newsroom 2026-04-22 21:57 UTC Key: openai-newsroom::identity::speeding-up-agentic-workflows-with-websockets Confidence: strong Mode: claude

Article body

When you ask Codex to fix a bug, it scans your codebase, reads relevant files, applies edits, and runs tests—all through dozens of back-and-forth API requests. Under the hood, each round-trip added latency that compounded across a single task. OpenAI's engineering team identified that as inference speed improved, the cumulative overhead of the Responses API itself was becoming the bottleneck rather than the model inference stage.

The solution was a persistent WebSocket connection. Instead of establishing a new HTTP connection and transmitting the full conversation history with every follow-up request, WebSocket mode lets clients send only new information while the server caches reusable state—including prior input and output items, tool definitions, and previously rendered tokens—in memory for the lifetime of the connection. By reusing that cached state and overlapping non-blocking postinference work like billing with subsequent requests, the team eliminated repeated API work across an entire agent rollout.

The results were significant. GPT-5.3-Codex-Spark, a coding-focused model, hit the target of over 1,000 tokens per second, with bursts reaching 4,000 TPS. External developers reported gains as well: Vercel's AI SDK saw up to 40% latency reduction, Cline's multi-file workflows ran 39% faster, and Cursor reported up to 30% faster model responses. WebSocket mode keeps the same `response.create` input shape developers already use, requiring only a `previous_response_id` to tap into the cached state.

Why this matters

  • As model inference gets faster, the infrastructure around it must keep pace; WebSocket mode closes that gap by removing redundant processing overhead that would otherwise negate speed gains at the model layer.
  • Developers don't need to rewrite their integration: the familiar API shape stays the same, making adoption incremental rather than a ground-up rearchitecting.
  • Third-party tools and IDEs—including Codex, Cursor, Cline, and Vercel—are already reporting real, measurable latency reductions in production workflows.
